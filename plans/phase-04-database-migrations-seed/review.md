# Review — Phase 04: Database, Migrations, and Seed

## Summary

Phase 04 adds the Payload CMS migration workflow and an idempotent development seed
script to the NexPress platform. The PostgreSQL adapter was already wired in Phase 03;
this phase formalizes the migration directory, adds all migration CLI scripts, and
creates the seed tooling with unit-level smoke tests.

No database schema changes were made beyond what Payload manages automatically
via its adapter. The initial migration file is generated by `pnpm migrate:create`
against a live database — this step requires a running PostgreSQL instance and is
documented in the runbook, not automated in CI (which has no live DB yet).

## Migration files created

No migration files are committed in this phase. The initial migration must be created
by running `pnpm migrate:create` against a live PostgreSQL instance. This is a
deliberate choice: committing a migration generated without a DB connection would
produce an empty/incorrect file. The runbook documents this step.

## Seed files/scripts created

- `apps/web/src/scripts/seed.ts` — TypeScript seed script using Payload Local API
- `apps/web/src/scripts/seed.test.ts` — 6 unit smoke tests for env-validation logic

Seed behavior:
- Requires `SEED_ADMIN_EMAIL` and `SEED_ADMIN_PASSWORD` from env
- Checks for existing user before creating (idempotent)
- Fails loudly with `process.exit(1)` on missing/invalid env
- Does NOT use `overrideAccess` — uses standard Payload collection create

## Security / access control notes

1. `DATABASE_URL` — never exposed to the client bundle; only accessed in Payload config
   at runtime and in the seed script (server-only `tsx` execution).
2. `SEED_ADMIN_EMAIL` / `SEED_ADMIN_PASSWORD` — env-only placeholders; no credentials
   in source code.
3. Seed script does not use `overrideAccess`. Payload's own collection-level access
   rules apply. The Users collection restricts reads to authenticated users.
4. `migrate:fresh` is documented as dev-only in the runbook. No guard is enforced in
   code — this is acceptable pre-production but should be gated in Phase 25.
5. Backup preconditions are documented in `docs/runbooks/migrations.md`.

## Known gaps

- No integration test against a real database (requires a live PostgreSQL instance in CI).
  Planned for a later phase when a CI database service is configured.
- The initial migration file must be manually created by a developer with DB access.
  This is expected and documented in the runbook.
- `migrate:fresh` has no environment guard yet. Phase 25 (security hardening) should
  add a `NODE_ENV !== 'production'` check or remove the script from production deploys.
- `process.exit(0)` in the seed script is a known Payload Local API constraint.
